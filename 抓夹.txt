### 第一步：验证“眼睛”是否睁开（获取相机图像）

我们需要确认 Gazebo 中的相机正在正确地向 ROS2 系统发送数据。

1. 在终端中，保持你的 `demo.launch.py` 继续运行。
2. 打开一个**新的终端**（别忘了 source 你的 ROS2 环境：`source /opt/ros/humble/setup.bash`）。
3. 运行图像可视化工具：
   ```bash
   ros2 run rqt_image_view rqt_image_view
   ```
4. 在弹出的窗口左上角的下拉菜单中，选择以下两个话题看看有没有画面：
   * `/camera/color/image_raw` （这应该是彩色的仿真世界画面）
   * `/camera/depth/image_raw` （这应该是黑白的深度图，距离越近颜色越深）

*(如果你在画面里只看到了灰蒙蒙的一片，试着在 Gazebo 里往机器人正前方放一个方块模型：点击 Gazebo 上方的 `Insert` 选项卡，随便拖一个 Box 到相机视野里)*

### 第二步：验证“手”是否听话（测试底层动作接口）

在让 MoveIt 进行复杂的 3D 路径规划之前，我们要确保底层的 `gripper_controller` 能够接收并执行最基础的“开合”指令。

在刚才的新终端里，直接通过命令行向抓夹发送一个“闭合”的动作目标（将 `finger_joint1` 移动到 `0.0` 的位置）：

```bash
ros2 action send_goal /gripper_controller/follow_joint_trajectory control_msgs/action/FollowJointTrajectory "{trajectory: {joint_names: ['finger_joint1'], points: [{positions: [0.0], time_from_start: {sec: 2, nanosec: 0}}]}}"
```

执行后，切回 RViz 和 Gazebo 的窗口，你应该能看到那两根虚拟的手指缓缓闭合！如果想让它再次张开，只需把上面命令里的 `[0.0]` 改成 `[0.03]` 即可。

### 下一步：正式编写“视觉 -> 抓取”代码

如果上面两步的测试都完美通过，说明硬件接口 100% 准备就绪。接下来我们就要进入 **LLM Grasping Capstone** 的核心代码迁移了。

这个项目将分为两个平行的代码节点（Node）：
1. **感知节点 (Vision Node - Python)：** 订阅刚才的相机图像，运行深度学习算法（如目标检测），计算出目标物体在 3D 空间中的坐标 $(x, y, z)$，并发布出去。
2. **执行节点 (MTC Node - C++)：** 订阅这个 3D 坐标，利用 MoveIt Task Constructor 生成“靠近 -> 张开 -> 下降 -> 闭合 -> 抬起”的连贯动作。
